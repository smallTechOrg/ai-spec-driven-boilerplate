# Data Model

All persistence uses SQLite (WAL mode). SQLAlchemy ORM with Alembic migrations manages schema evolution.

---

## Storage Technology

SQLite with WAL (Write-Ahead Logging) mode for concurrent reads during write operations. The database file is stored at the path configured by `AGENT_DATABASE_URL` (default: `sqlite:///./data/agent.db`). This is the only storage engine in Phase 1 through Phase 3; PostgreSQL is deferred to Phase 4.

---

## Entities

### `runs` (extended from skeleton)

Stores every invocation — one row per `POST /sessions/{id}/analyze` call.

| Field | Type | Required | Description |
|-------|------|----------|-------------|
| `id` | TEXT (UUID) | Yes | Primary key; auto-generated |
| `status` | TEXT | Yes | `pending`, `completed`, or `failed` |
| `input_text` | TEXT | No | Raw user question (backward compat alias for `question`) |
| `output_text` | TEXT | No | Prose narrative from the Insights node |
| `error_message` | TEXT | No | Error string when status = `failed` |
| `created_at` | TIMESTAMP WITH TIME ZONE | Yes | Set on insert |
| `updated_at` | TIMESTAMP WITH TIME ZONE | Yes | Set on insert and update |
| `session_id` | TEXT (UUID) | No | FK to `sessions.id`; links run to a session |
| `question` | TEXT | No | Natural-language question verbatim |
| `sql_query` | TEXT | No | SQL generated by the nl-to-sql node |
| `insight_json` | TEXT (serialized JSON) | No | Structured statistics from the Insights node |
| `chart_specs` | TEXT (serialized JSON) | No | JSON array of ChartSpec objects |

Indexes: `(session_id)` non-unique index for listing a session's runs.

---

### `sessions`

Groups related questions and their uploaded tables into one logical workspace.

| Field | Type | Required | Description |
|-------|------|----------|-------------|
| `id` | TEXT (UUID) | Yes | Primary key; auto-generated |
| `created_at` | TIMESTAMP WITH TIME ZONE | Yes | Set on insert |
| `updated_at` | TIMESTAMP WITH TIME ZONE | Yes | Set on update |

---

### `uploaded_files`

Tracks every file ingested into SQLite for a session.

| Field | Type | Required | Description |
|-------|------|----------|-------------|
| `id` | TEXT (UUID) | Yes | Primary key; auto-generated |
| `session_id` | TEXT (UUID) | Yes | FK to `sessions.id` |
| `filename` | TEXT | Yes | Original uploaded filename including extension |
| `table_name` | TEXT | Yes | SQLite table name derived from filename |
| `row_count` | INTEGER | Yes | Row count after ingest |
| `column_names` | TEXT (serialized JSON array) | Yes | List of sanitized column names |
| `created_at` | TIMESTAMP WITH TIME ZONE | Yes | Set on insert |

Indexes: `(session_id)` for listing session files.

---

### `analysis_cache`

Caches query execution results to avoid duplicate LLM calls.

| Field | Type | Required | Description |
|-------|------|----------|-------------|
| `id` | TEXT (UUID) | Yes | Primary key; auto-generated |
| `question_hash` | TEXT | Yes | `sha256(question.strip().lower())` hex digest |
| `table_hash` | TEXT | Yes | `sha256(sorted(table_name:row_count pairs))` hex digest |
| `result_json` | TEXT (serialized JSON) | Yes | Cached payload: `{sql_query, query_rows, insight_json, chart_specs}` |
| `created_at` | TIMESTAMP WITH TIME ZONE | Yes | Set on insert |

Unique constraint: `(question_hash, table_hash)`.

---

### Dynamic data tables (not ORM-managed)

Each uploaded file creates a SQLite table named after the sanitized filename. These are created/replaced by pandas `to_sql(if_exists="replace")`. They are not tracked by Alembic migrations and have no ORM models. Schema is introspected at query time via `PRAGMA table_info(<table_name>)`.

---

## Relationships

- `sessions` 1 → N `uploaded_files` (via `uploaded_files.session_id`)
- `sessions` 1 → N `runs` (via `runs.session_id`)
- `analysis_cache` is keyed by content hash — no FK to sessions or runs

---

## Data Lifecycle

1. `POST /sessions` — creates a `sessions` row.
2. `POST /sessions/{id}/files` — creates an `uploaded_files` row and a dynamic SQLite table. Re-upload of same filename drops and recreates the dynamic table and replaces the `uploaded_files` row.
3. `POST /sessions/{id}/analyze` — creates a `runs` row (`status=pending`), invokes the agent graph, updates the row to `status=completed` with all result fields. On cache hit: row is still created for history but result fields are copied from `analysis_cache.result_json`.
4. Data is never automatically deleted. Local-only; no cloud synchronization.

---

## Sensitive Data

- No PII is collected. Uploaded CSV/Excel data is the user's own local data and stays entirely on disk.
- The `AGENT_GEMINI_API_KEY` is stored only in `.env` (gitignored); it is never persisted to the database.
- No other secrets are stored.

## Alembic Migration Plan

- `0001_initial.py` — existing skeleton (`runs` table with base columns).
- `0002_data_analysis.py` — new migration that adds `sessions`, `uploaded_files`, `analysis_cache` tables; adds `session_id`, `question`, `sql_query`, `insight_json`, `chart_specs` columns to `runs`.
