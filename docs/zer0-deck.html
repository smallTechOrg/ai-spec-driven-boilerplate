Zer0

A four-layer coding-agent harness. Zero-shot from intent to a working system — the empty ground a build takes form from.

spec (goal) → src (action) → logs (outcome) → harness reconciles → spec …

What this deck is: the theory, what's already built (PR restructure/zer0), the open questions with my recommendations, and the academic way forward. Arrow keys / Space to move.

The thesis

Most "AI builds your app" tools generate code and stop. Zer0's claim: a build is not done when the code runs — it's done when the outcome matches the goal, and you can prove they reconcile.

Three sources of truth

spec/ = the goal · src/ = the action · logs/ = the outcome. Each is true about the system from a different angle.

One practice over them

harness/ is mindfulness — it checks goal vs action vs outcome and adjusts. A verb, not an artifact.

Agent-agnostic by design: harness/ is the protocol; .claude/ is just the binding to one runtime.

Drift table: spec≠src → fix code · src≠logs → fix bug · logs≠spec → fix code, or amend the goal if the goal was wrong.

The team — always-on roles

manager · main session

Coordinates the team, holds the whole picture, owns the human gate and adjustments.

designer · the goal

Turns your prompts into a complete spec (line-level), designs UX. PM + UX + author.

engineer · the action

Feasibility on every draft, then builds to the approved spec — no more, no less.

qa · the bar

Reviews spec & code, owns tests AND the end-user's goals. "Nothing gets past qa."

analyst · the outcome → the loop

Always observing logs, traces, tests, timings, and your prompts & frustrations. Feeds reality back to everyone; proposes spec amendments when the goal itself is wrong. This role is the feedback loop.

The build lifecycle

INTAKE ─▶ BUILD ─▶ RECONCILE ─▶ SHIP (designer) (engineer) (analyst) (manager)
  • Intake — designer interviews you as long as it takes, to the line level; engineer checks feasibility on each draft. Gate = completeness checklist passes and designer + engineer + qa sign off. The one human gate.
  • Build — phases sliced by end-user value, derived per spec; each runs end-to-end and passes its test before the next.
  • Reconcile — analyst checks outcome vs goal at every gate; writes logs/analysis/; proposes spec amendments when the goal was wrong.
  • Ship — finalize the PR; deploy is a later phase (Render default).
